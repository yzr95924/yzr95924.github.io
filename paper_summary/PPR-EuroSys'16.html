1. Motivation

Reducing repair traffic without negatively impacting storage overhead and data reliability is a challenging task. (How to do the tradeoff between two ends?)

Tow ends:

  1. Minimum Storage Codes: minimum storage overhead, high repair bandwidth
  2. Minimum Bandwidth Codes: optimal repair traffic, high storage overhead

This paper wants to design a practical EC repair technique that reduces repair time without negatively affecting data reliability, storage overhead, and repair degree. (Essence: reduce repair time but not the total repair traffic aggregated over the links.)

2. Overview

  • Main Idea of this paper This paper deems that poor utilization of network resources is the main reason why reconstruction is slow. Specifically, the network link into the repair server.

The process of conveying all the chunks to a single point in itself creates a bottleneck and load imbalance on some network links.

This paper solve this problem by redistributing the reconstruction traffic more uniformly across the existing network links. Thus, it can improve the utilization of network resources and decreasing reconstruction time.

PPR divides reconstruction into several partial parallel repair operations that are performed simultaneously at multple server.

Theoretically, PPR can complete the network transfer for a single chunk reconstruction in time, compared to time needed for a RS code.

Partial Parallel Repair (PPR)

This method wants to reduce the pressure on the primary constrained resources: network capacity and disk reads. PPR divides the entire repair operation into a set of partial operations that ** are then scheduled to execute in parallel on multiple servers**.

the ground true of PPR: 1) Reconstruction equation is linear and associative. (the reconstruction equation can be decomposed into a few independent and partial associative operations) 2) All individual terms in reconstruction equation will have the same volume of data (equal to the chunk size)

Multiple Concurrent Repairs: m-PPR

Performing multiple overlapping reconstructions may put pressure on shared resources, e.g., network and disk I/O, which can lead to poor reconstruction performance.

The idea of m-PPR is to schedule multiple reconstruction-jobs in parallel while trying to minimize the competition for shared resources between multiple reconstruction operation. (Using the Repair Manager(RM), e.g. Meta-Server)

RM needs to keep track of various information for all the servers and uses greedy heuristics to choose the best source and destination servers for each reconstruction job by calculating the source weight and the destination weight.

These source and destination weights represent the goodness of a server as a source or destination candidate for scheduling the next repair job.

  • Implementation Where: Do the implementation based on QFS, change the codebase of QFS. RM calculates decoding coefficients corresponding to each participating chunk, and then distributes these coefficients along with a repair plan.

Degrader Reads: a Client acts as the repair site and informs the RM about the missing chunk. Regular Repairs: RM choose a repair site.

The RM uses nodes usage statistics to de-prioritize the slow nodes before creating the repair plan.

3. Strength and Weakness

The Strengths:

  1. This work is orthogonal to the coding scheme and placement techniques so that it is can be overlaid on top of almost all published erasure coding schemes. (easy to extend)
  2. PPR provides chances of Parallel Computations and Reduced Memory Footprint, compared with traditional RS reconstruction.
  3. It also considers the case that there are multiple concurrent repairs and provides a scheduled algorithm to minimize the competition for shared resources.

The Weaknesses:

  1. How to identify the time step in heterogeneous environment? It does not consider this question in details.
  2. As RM needs to collect the information for each server (using heartbeat), it needs to guarantee the stale of information. This fact will introduce more frequent heartbeat message. (high overhead, especially there are more than one RMs)
  3. The load of each link is not absolutely uniform (that is the point of ECPipe to solve)

4. How to extend this paper

  1. I suppose this paper can be extended by considering how to choose repair site with heterogeneous environment. Currently, this paper just consider a very simple case.
  2. When there are multiple RM, how to reduce the overhead of them is another question that should be considered.
